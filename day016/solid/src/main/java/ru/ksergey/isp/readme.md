# Interface segregation principle

Принцип разделения интерфейсов гласит, что клиенты не должны зависеть от интерфейсов, которые они не используют. Интерфейсы должны быть разделены на более мелкие, специфические части, чтобы минимизировать зависимости.

Плохой пример

```JAVA
public interface Worker {
    void work();
    void eat();
    void sleep();
}

public class Manager implements Worker {
    @Override
    public void work() {
        // Реализация работы менеджера
    }

    @Override
    public void eat() {
        // Реализация приема пищи менеджера
    }

    @Override
    public void sleep() {
        // Реализация сна менеджера
    }
}

public class Robot implements Worker {
    @Override
    public void work() {
        // Реализация работы робота
    }

    @Override
    public void eat() {
        throw new UnsupportedOperationException("Robots cannot eat.");
    }

    @Override
    public void sleep() {
        throw new UnsupportedOperationException("Robots cannot sleep.");
    }
}
```
В этой плохой реализации интерфейс `IWorker` имеет методы, которые не имеют смысла для всех его реализующих классов. Например, методы `Eat` и `Sleep` не имеют смысла для класса `Robot`, но им приходится реализовывать и выбрасывать исключения.

Хороший пример

```JAVA
public interface Worker {
    void work();
}

public interface Eater {
    void eat();
}

public interface Sleeper {
    void sleep();
}

public class Manager implements Worker, Eater, Sleeper {
    @Override
    public void work() {
        // Реализация работы менеджера
    }

    @Override
    public void eat() {
        // Реализация приема пищи менеджера
    }

    @Override
    public void sleep() {
        // Реализация сна менеджера
    }
}

public class Robot implements Worker {
    @Override
    public void work() {
        // Реализация работы робота
    }
}

```

В этой хорошей реализации интерфейсы разделены на более мелкие, специфические части, что позволяет классу `Robot` не реализовывать несмысленные методы `Eat` и `Sleep`, а классу `Manager` реализовать только те методы, которые ему действительно нужны.

## Техническое задание: Описание автомобилей

### Описание:
Разработать систему для моделирования и описания различных моделей автомобилей, с акцентом на марки Audi и Lada. Система должна обеспечивать гибкое описание характеристик и комплектаций автомобилей, используя принципы объектно-ориентированного программирования и интерфейсов.

### Требования:

1. **Базовая структура:**
   - Абстрактный класс `Car` как основа для всех автомобилей.
   - Классы `Audi` и `Lada`, наследующие от `Car`.
   - Классы `AudiA5` и `VestaImpl`, представляющие конкретные модели.

2. **Характеристики:**
   - Класс `Characteristic` для хранения и управления отдельными характеристиками.

3. **Интерфейсы для комплектаций:**
   - Набор интерфейсов для различных пакетов комплектаций (например, `SLine`, `Infotainment`, `AudiPreSenseFront`).
   - Составные интерфейсы для комбинированных пакетов (например, `AdvanceLine`, `AudiExclusive`).

4. **Специфические требования для Vesta:**
   - Интерфейс `Vesta` с методами для описания характеристик модели Vesta.
   - Классы `Vesta113hp` и `Vesta122hp`, реализующие `Vesta` с различными характеристиками.

5. **Реализация моделей:**
   - Класс `MyAudiA5`, реализующий интерфейс `AudiExclusive`.
   - Классы `Vesta113hp` и `Vesta122hp` с реализацией всех методов из `Vesta`.

### Дополнительные требования:

1. Обеспечить соблюдение принципа разделения интерфейса (ISP).
2. Реализовать возможность легкого добавления новых моделей и характеристик.
3. Использовать фабричный метод `set` в классе `Characteristic` для создания объектов.

### Примечания:

- Система должна быть расширяемой для добавления новых марок и моделей автомобилей.
- Рассмотреть возможность использования паттерна Строитель для сложных конфигураций автомобилей.
- Обеспечить корректную обработку неподдерживаемых характеристик (например, с использованием исключений).
- Продумать механизм для эффективного управления большим количеством характеристик и комплектаций.
