# CRUD и CRUD-репозиторий
CRUD - это акроним, обозначающий четыре базовые функции, используемые при работе с базами данных и персистентными хранилищами данных:
- **C**reate (создание)
- **R**ead (чтение)
- **U**pdate (обновление)
- **D**elete (удаление)

В SQL этим операциям соответствуют:
- Insert - для создания записей
- Select - для чтения записей  
- Update - для редактирования записей
- Delete - для удаления записей

Хотя традиционно подход CRUD применяется к базам данных, он может быть распространён на любые хранимые вычислительные сущности - файлы, структуры в памяти, объекты.

Преимущества использования модели CRUD:
- **Стандартизация**: создаёт общую основу для взаимодействия с системами хранения данных
- **Упрощение разработки**: предоставляет чёткий набор операций для манипулирования данными
- **Улучшенная поддерживаемость**: приложения на основе CRUD легче поддерживать благодаря стандартизированному подходу
- **Улучшенный пользовательский опыт**: позволяет пользователям интуитивно выполнять ключевые задачи по работе с данными
- **Модульность и гибкость**: способствует модульности, позволяя легко изменять части приложения
- **Масштабируемость**: облегчает масштабирование приложений для соответствия растущим требованиям 

CRUD-репозиторий - это реализация шаблона Repository, предоставляющая обобщённый интерфейс для выполнения CRUD операций с различными базами данных.

# DAO (Data Access Object) 
Это паттерн проектирования, обеспечивающий абстрактный интерфейс для доступа к данным. Он отделяет низкоуровневые операции с базой данных от бизнес-логики приложения. 

DAO инкапсулирует логику взаимодействия с хранилищем данных, предоставляя унифицированный интерфейс для CRUD-операций. Это позволяет легко менять механизм хранения данных без изменения остальных частей приложения. 

DAO улучшает структуру кода, повышает его переносимость и упрощает тестирование. Типичный интерфейс DAO включает методы для получения, создания, обновления и удаления объектов. 

# Слоёная архитектура

- Архитектурный паттерн, разделяющий приложение на логические слои
- Каждый слой имеет свою ответственность и инкапсулирует связанную функциональность
- Слои взаимодействуют друг с другом через чётко определённые интерфейсы
- Изменения в одном слое минимально влияют на другие слои

### Типичные слои

1. **Слой представления (Presentation Layer)**
   - Отвечает за взаимодействие с пользователем
   - Включает пользовательский интерфейс, обработку ввода и вывода
2. **Слой бизнес-логики (Business Logic Layer)**
   - Содержит основную бизнес-логику приложения
   - Координирует работу приложения, обрабатывает данные
3. **Слой доступа к данным (Data Access Layer)**
   - Обеспечивает доступ к данным из постоянного хранилища
   - Включает компоненты для работы с базой данных, файлами, внешними сервисами

### Преимущества слоёной архитектуры

- Разделение ответственности и модульность
- Повышение поддерживаемости и тестируемости
- Возможность независимой разработки и развёртывания слоёв
- Гибкость и адаптируемость к изменениям требований

# Для чего нужны DTO-модели

DTO (Data Transfer Object) модели используются для передачи данных между различными слоями или компонентами приложения. Основные цели применения DTO:

## Инкапсуляция и абстракция

DTO позволяют инкапсулировать данные, передаваемые между слоями, и абстрагировать внутреннее представление данных от внешнего интерфейса. Это обеспечивает слабую связанность (loose coupling) между компонентами.

## Уменьшение количества вызовов

Вместо множества вызовов методов с примитивными параметрами, можно передать один объект DTO, содержащий все необходимые данные. Это сокращает количество удалённых вызовов и повышает производительность.

## Унификация форматов данных

DTO стандартизируют форматы данных, которыми обмениваются компоненты системы. Каждый слой может иметь свои внутренние модели данных, оптимизированные под его нужды, а для коммуникации использовать единый формат DTO.

## Версионирование интерфейсов

DTO позволяют организовать версионирование интерфейсов передачи данных между компонентами. При изменении формата обмена старые клиенты могут продолжать использовать старую версию DTO, а новые - перейти на новую версию.

## Примеры использования DTO

- Передача данных между клиентом и сервером через API
- Обмен данными между микросервисами или слоями приложения (например, между контроллером, сервисом, репозиторием) 

## Техническое задание на создание сервиса управления контактами

### 1. Введение
Целью данного проекта является создание сервиса для управления контактами, который позволит пользователям хранить, добавлять, удалять и обновлять информацию о контактах.

### 2. Функциональные требования
Сервис должен предоставлять следующие функциональные возможности:
1. Получение списка всех контактов.
2. Получение контакта по его идентификатору.
3. Добавление нового контакта.
4. Удаление контакта по его идентификатору.
5. Обновление информации о контакте по его идентификатору.

### 3. Архитектура
Сервис должен быть реализован с использованием следующих компонентов:
- Интерфейс `Storage`, определяющий методы для работы с хранилищем контактов.
- Класс `InMemoryStorage`, реализующий интерфейс `Storage` и хранящий контакты в оперативной памяти.
- Класс `Contact`, представляющий модель контакта с полями: идентификатор, имя и email.
- Класс `ContactDto`, представляющий модель для передачи данных о контакте при обновлении.

### 4. Методы
#### 4.1. Получение списка контактов
- Метод: `findAll()`
- Возвращает список всех контактов из хранилища.

#### 4.2. Получение контакта 
- Метод: `findById(int id)`
- Возвращает контакт с указанным идентификатором если он есть из хранилища, если нет - возвращать `null`.

#### 4.3. Добавление контакта
- Метод: `save(Contact contact)`
- Принимает объект `Contact` для добавления в хранилище.
- Возвращает добавленный объект `Contact` или `null`, если контакт с таким идентификатором уже существует.

#### 4.4. Удаление контакта
- Метод: `delete(int id)`
- Принимает идентификатор контакта для удаления из хранилища.
- Возвращает `true`, если контакт успешно удален, иначе `false`.

#### 4.5. Обновление контакта
- Метод: `updateContact(ContactDto contactDto, int id)`
- Принимает объект `ContactDto` с обновленными данными и идентификатор контакта для обновления.
- Обновляет поля `name` и `email` контакта, если соответствующие поля в `ContactDto` не пустые.
- Возвращает `true`, если контакт успешно обновлен, иначе `false`.

### 5. Реализация
- Сервис должен быть реализован на языке Java.
- Хранилище контактов должно быть реализовано в классе `InMemoryStorage`, который хранит контакты в списке в оперативной памяти.
- Изначально хранилище должно быть заполнено некоторыми тестовыми данными.
